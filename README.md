# Mystery Mansion Online

Учебная веб-адаптация настольной детективной игры в стиле Cluedo.

Проект находится в стадии активной разработки: базовый игровой цикл уже работает, добавлена система JSON-карт, валидатор карт, визуальный просмотрщик карт и секретные проходы.

---

## 1. Цель проекта

Создать стабильную браузерную многопользовательскую игру, в которой игроки:

- создают лобби;
- выбирают персонажа;
- ходят по игровому полю;
- входят в комнаты;
- делают предположения;
- опровергают предположения картами;
- используют секретные проходы;
- делают финальное обвинение;
- завершают партию победой или поражением.

Долгосрочная цель — сделать игру расширяемой: новые карты, визуальные наборы, карточки, админка карт и потенциально пользовательские карты.

## 4. Запуск проекта

1. Скопировать папку `cluedo_web` в `htdocs` или папку локального PHP-сервера.
2. Создать базу данных:

```sql
CREATE DATABASE cluedo_web CHARACTER SET utf8mb4 COLLATE utf8mb4_unicode_ci;
```

3. Импортировать SQL-файл:

```text
sql/cluedo_web.sql
```

4. Проверить доступ к базе данных в файле:

```text
includes/config.php
```

Пример настроек:

```php
const DB_NAME = 'cluedo_web';
const DB_USER = 'root';
const DB_PASS = '';
```

5. Открыть проект в браузере:

```text
http://localhost/cluedo_web/
```

## 6. Игровой процесс

Основная логика игры проходит через `api.php`.

Ключевые API-действия:

- `state` — получить текущее состояние игры;
- `start` — начать игру;
- `roll` — бросить кубики;
- `move` — переместиться;
- `secretPassage` — использовать секретный проход;
- `suggest` — сделать предположение;
- `showCard` — показать карту при опровержении;
- `accuse` — сделать финальное обвинение;
- `endTurn` — завершить ход;
- `surrender` — сдаться.

---

## 7. Фазы игры

В игре используются фазы:

```text
join
roll
move
suggest
disprove
accuse
ended
```

Краткая схема:

```text
join
  ↓ start
roll
  ↓ roll
move
  ↓ вход в комнату
suggest
  ↓ suggest
disprove / accuse
  ↓ showCard / no cards
accuse
  ↓ endTurn
roll следующего игрока
```

Секретный проход используется в фазе `roll` вместо броска кубиков. После использования прохода игра переходит в фазу `suggest`.

## 8. JSON-карты

Карты хранятся в папке:

```text
maps/
```

Сейчас в проекте есть карты:

```text
classic_mansion.json
mansion_shifted_doors.json
mansion_evening.json
mansion_crossroads.json
```

Карта описывает:

- `id` — технический идентификатор карты;
- `title` — название карты;
- `description` — описание;
- `variant` — числовой вариант;
- `board` — размер поля;
- `paths` — клетки-коридоры;
- `starts` — стартовые позиции персонажей;
- `rooms` — комнаты, двери, secret-проходы и темы.

Подробное описание формата JSON-карт вынесено в отдельный файл с документацией по картам.

## 9. Валидатор карт

Файл:

```text
tools/validate_maps.php
```

Запуск:

```bash
php tools/validate_maps.php
```

Валидатор проверяет:

- валидность JSON;
- совпадение `id` с именем файла;
- наличие `title`;
- размеры поля;
- обязательные комнаты;
- координаты комнат;
- пересечение комнат;
- выход комнат за границы поля;
- двери;
- соседнюю клетку коридора у двери;
- достижимость дверей;
- структуру `paths`;
- выход `paths` за границы поля;
- `paths` внутри комнат;
- связность `paths`;
- изолированные коридоры;
- secret-проходы;
- стартовые позиции персонажей;
- итоговую статистику карты.

Пример успешного вывода:

```text
classic_mansion.json: OK
  [INFO] board=17x17, rooms=9, paths=70, doors=9, secrets=4, starts=6, explicit_paths=yes
```

## 11. Анализ расстояний

В preview есть минимальный анализ карты.

Он считает расстояния от стартовых позиций до дверей комнат:

- минимальную дистанцию;
- среднюю дистанцию;
- максимальную дистанцию;
- возможный перекос.

Важно:

```text
Валидатор отвечает на вопрос: карта технически рабочая?
Анализатор отвечает на вопрос: есть ли подозрительный дисбаланс?
```

Жёлтое предупреждение анализатора не означает, что карта сломана. Оно означает, что карту нужно дополнительно проверить вручную.

## 12. Секретные проходы

Секретные проходы хранятся в JSON-карте в поле `secret` у комнаты.

Пример:

```json
"Кухня": {
  "secret": "Кабинет"
},
"Кабинет": {
  "secret": "Кухня"
}
```

Правила работы:

- проход доступен только в начале хода;
- технически это фаза `roll`;
- игрок выбирает между броском кубиков и secret-проходом;
- после прохода игра переходит в `suggest`;
- игрок может сделать предложение или обвинение.

---

## 13. Текущее ограничение карт

Сейчас разные карты могут менять:

- размер поля;
- комнаты;
- двери;
- коридоры;
- secret-проходы;
- стартовые позиции;
- визуальные темы.

Пока не рекомендуется менять:

- набор комнат;
- набор персонажей;
- набор оружия.

Причина: текущая карточная логика ещё использует фиксированные текстовые названия.

Будущий этап — переход на систему карточек с внутренними ID.

---

## 14. Будущая система карточек

Текущая логика использует строки:

```text
Кухня
Кинжал
Алекс Громов
```

В будущем планируется перейти к карточной модели:

```json
{
  "id": "room_kitchen",
  "type": "room",
  "title": "Кухня",
  "image": "assets/cards/rooms/kitchen.png"
}
```

Это позволит:

- отделить внутреннюю игровую логику от отображаемого текста;
- добавлять изображения карточек;
- делать разные наборы комнат, персонажей и оружия;
- поддерживать тематические карты;
- легче добавлять event-наборы.

---

## 15. Важные технические заметки

- `characters()` пока хранит список персонажей, их имена и цвета.
- Стартовые позиции персонажей уже вынесены в JSON-карты через `starts`.
- `rooms()` пока остаётся частью старой карточной логики.
- `weapons()` и `suspects()` также пока остаются статическими.
- `available_maps()` пока вручную определяет карты, доступные в лобби.
- Tools-страницы сейчас являются dev-инструментами и в будущем должны быть защищены.
